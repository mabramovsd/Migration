## 🏝️ Migration Island
Пет-проект-симулятор замкнутой экосистемы острова для отработки микросервисной архитектуры, миграций Entity Framework Core и работы с RabbitMQ.

# Описание
Представьте себе необитаемый остров, на котором со временем появились три ключевых предприятия: Сельское хозяйство, Судостроительство и Энергетика. На острове живет ограниченное количество людей (например, 100 человек), которые могут свободно перемещаться между предприятиями.

Этот проект — архитектурный тренажер. Его цель — не просто написать код, а спроектировать систему из независимых сервисов, которые обмениваются данными через шину сообщений (RabbitMQ). Это идеальная песочница для отработки паттернов Event-Driven и CQRS без риска сломать реальный продакшн-код.

# Архитектура
Проект построен на принципах микросервисной архитектуры:
- Разделение ответственности (Bounded Contexts): Каждый аспект бизнеса вынесен в отдельный сервис (Workforce, Agriculture, Shipbuilding).
- Независимость данных: У каждого сервиса своя база данных (в рамках одной СУБД, но в разных схемах или таблицах), к которой нет прямого доступа у других сервисов.
- Асинхронное взаимодействие: Сервисы общаются друг с другом исключительно через события (Events), публикуемые в общую очередь RabbitMQ. Например, при увольнении сотрудника из Сельского хозяйства, WorkforceService публикует событие, которое может быть обработано другими сервисами.
- Единые контракты: Все сервисы используют общие DTO и интерфейсы, описанные в отдельной библиотеке контрактов.
- Агрегация данных: Для вывода информации на дашборд используется подход API Composition, где фронтенд собирает данные с эндпоинтов всех сервисов.

# Технологии
- Язык и фреймворк: C#, .NET 8 (или выше), ASP.NET Core Web API.
- Клиентская часть: SPA-приложение (React/Vue/Blazor WASM).
- Шина сообщений: RabbitMQ.
- СУБД: MS SQL Server (используется одна БД с несколькими схемами для изоляции данных сервисов).
- ORM: Entity Framework Core (для работы с базой данных и миграциями).
- Инъекция зависимостей: Встроенный DI-контейнер .NET.

# Запуск
Примечание: Проект настроен на работу с локальными инстансами сервисов. Docker не требуется.

Предварительные требования:
- Установленный MS SQL Server (LocalDB или Developer Edition).
- Установленный RabbitMQ.
- .NET SDK версии 8.0 или выше.

Шаги запуска:
- Запустите зависимости
- Убедитесь, что службы MS SQL Server и RabbitMQ запущены.
- Создайте базу данных
- Откройте терминал в папке Migration.Database (если вы выделили ее) или используйте стандартный инструмент управления вашей СУБД и создайте пустую базу данных (например, MigrationIslandDb).
- Примените миграции (bash)
    - **Перейдите в директорию любого сервиса, использующего EF Core (например, Workforce)**
    - cd src/Migration.Workforce
    - **Добавьте начальную миграцию**
    - dotnet ef migrations add InitialCreate --context ApplicationDbContext
    - **Обновите базу данных**
    - dotnet ef database update --context ApplicationDbContext
    - Повторите шаги для остальных сервисов, если они используют свои контексты.
- Запустите сервисы (В разных терминалах запустите каждый сервис, bash):
    - **Терминал 1**
    - cd src/Migration.Workforce
    - dotnet watch run
    - **Терминал 2**
    - cd src/Migration.Agriculture
    - dotnet watch run
    - **Терминал 3**
    - cd src/Migration.Shipbuilding
    - dotnet watch run
    - **Терминал 4**
    - cd src/Migration.Web
    - dotnet watch run
- Откройте приложение
- Дашборд будет доступен по адресу http://localhost:5000 (порт может отличаться, смотрите вывод консоли).

# Инструкция
Все действия в системе инициируются через API Gateway (Migration.Web).
- Вы совершаете действие на фронте (например, "Перевести сотрудника").
- Фронтенд отправляет HTTP-запрос на соответствующий контроллер в Migration.Web.
- Контроллер вызывает метод нужного сервиса (например, _workforceService.TransferEmployee(...)).
- Внутри сервиса происходит бизнес-логика и изменение состояния только своей базы данных.
- После успешного изменения сервис публикует доменное событие (например, EmployeeTransferredEvent) в RabbitMQ.
- Другие сервисы, которым интересна эта информация, подписаны на данное событие. Они получают сообщение, обрабатывают его и обновляют свое состояние.
- Фронтенд запрашивает актуальные данные у всех сервисов и отображает сводную информацию на дашборде.

# Зачем
- Практический опыт с микросервисами: Научитесь разбивать монолитную логику на независимые сервисы.
- Отработка миграций EF Core: Безопасно экспериментируете с добавлением, изменением и удалением колонок и таблиц.
- Понимание Event-Driven подхода: Увидите на практике преимущества и сложности асинхронного взаимодействия.
- Архитектурный шаблон: Получите готовый пример реализации Outbox-паттерна и разделения баз данных.
- Портфолио: Демонстрационный проект, который можно показать на собеседовании, чтобы доказать владение современными технологиями .NET.
